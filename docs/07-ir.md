## IR Fundamentals

Infrared (IR) communication uses **invisible light** just beyond the visible red spectrum to transmit data. It is the technology behind virtually every TV remote, air conditioner controller, soundbar remote, and many other consumer electronics controls.

### Physical Properties

| Property | Value |
|:---|:---|
| Wavelength | 850--950 nm (typically **940 nm** for consumer remotes) |
| Carrier Frequency | **38 kHz** (most common), also 36 kHz, 40 kHz, 56 kHz |
| Modulation | On-Off Keying (OOK) on carrier |
| Range | 1--10 m (typical consumer remote) |
| Beam Angle | 15--45 degrees (LED dependent) |
| Medium | **Line of sight** required (reflects off walls/ceilings) |
| Speed | 300 bps -- 4 Mbps (protocol dependent; remote controls are very low speed) |

### How IR Transmission Works

```
                    IR LED on T-Embed
                          │
                          ▼
    Carrier (38 kHz)     Modulated Signal        IR Receiver
    ┌┐┌┐┌┐┌┐┌┐┌┐        ┌┐┌┐┌┐    ┌┐┌┐┌┐      ┌─────────┐
    │││││││││││││   →    ││││││    ││││││  →    │ ┌─┐     │
    ┘└┘└┘└┘└┘└┘└┘        ┘└┘└┘└    ┘└┘└┘└       │ │●│ AGC │ → Demodulated
                                                │ └─┘     │    digital
    Data (NEC):          Combined:              │ Bandpass │    signal
    ────┐    ┌──         ┌┐┌┐┌┐    ┌┐┌┐┌┐      │ Demod   │
        │    │           ││││││    ││││││       └─────────┘
        └────┘           ┘└┘└┘└    ┘└┘└┘└
    mark  space          carrier   carrier
    (ON)  (OFF)          burst     burst
```

The IR LED does not simply turn on and off. Instead, it **blinks at the carrier frequency** (typically 38 kHz, which is ~26.3 microseconds per cycle) during "mark" periods, and remains off during "space" periods. The IR receiver module contains:

1. **Photodiode**: Detects IR light.
2. **Bandpass filter**: Tuned to the carrier frequency (rejects ambient IR from sunlight, lamps).
3. **AGC (Automatic Gain Control)**: Normalizes signal strength.
4. **Demodulator**: Strips the carrier, outputting a clean digital signal.

This is why IR remotes work reliably despite ambient infrared sources — the 38 kHz modulation acts as a signature that separates intentional signals from noise.

## IR vs. RF

Understanding the differences helps you choose the right tool for the job:

| Aspect | Infrared (IR) | Radio Frequency (RF) |
|:---|:---|:---|
| Medium | Light (photons) | Electromagnetic waves |
| Line of Sight | **Required** | Not required (passes through walls) |
| Range | 1--10 m typical | 10 m to several km |
| Interference | Sunlight, fluorescent lights | Other RF devices, Wi-Fi, noise |
| Security | Physical proximity needed, harder to intercept remotely | Can be intercepted from distance |
| Directionality | Highly directional (narrow beam) | Omnidirectional or directional (antenna dependent) |
| Cost | Very cheap (LED + photodiode) | More complex (radio transceiver) |
| Regulation | Unregulated | Licensed or ISM bands, regulated by FCC/ETSI |
| Typical Use | TV remotes, AC controls | Garage doors, car keys, smart home |
| Replay Complexity | Simple (no rolling codes) | Varies (fixed code to rolling code to encrypted) |

> IR signals are inherently more secure against remote interception because an attacker must have line of sight to the transmission. However, IR protocols almost never implement encryption or rolling codes, making replay attacks trivially easy for anyone within physical proximity.
{: .note }

## How IR Remotes Work

Every button press on an IR remote control follows this sequence:

```
Button Press → Encode → Modulate → Transmit → Receive → Demodulate → Decode → Action

Step-by-step:
┌──────────┐   ┌──────────┐   ┌──────────┐   ┌──────────┐
│ 1. User  │──→│ 2. MCU   │──→│ 3. IR LED│──→│ 4. Light │
│ presses  │   │ encodes  │   │ transmits│   │ travels  │
│ button   │   │ protocol │   │ modulated│   │ to target│
└──────────┘   └──────────┘   └──────────┘   └──────────┘
                                                   │
┌──────────┐   ┌──────────┐   ┌──────────┐        │
│ 7. TV    │←──│ 6. MCU   │←──│ 5. IR    │←───────┘
│ changes  │   │ decodes  │   │ receiver │
│ channel  │   │ command  │   │ demod.   │
└──────────┘   └──────────┘   └──────────┘
```

### Encoding Concepts

All IR protocols encode binary data using variations of **pulse timing**:

- **Pulse Distance Encoding** (NEC, Samsung): The duration of the space (off time) between bursts determines whether a bit is 0 or 1.
- **Manchester / Bi-phase Encoding** (RC5, RC6): A transition in the middle of the bit period encodes the value.
- **Pulse Width Encoding** (Sony SIRC): The duration of the mark (on time) determines the bit value.

## IR Protocols Deep Dive

### NEC Protocol (Most Common)

The NEC protocol is used by the majority of consumer electronics. If you capture an IR signal and do not know the protocol, it is most likely NEC.

**Specifications:**

| Parameter | Value |
|:---|:---|
| Carrier | 38 kHz |
| Encoding | Pulse Distance |
| Bits | 32 (8-bit address + 8-bit inverse address + 8-bit command + 8-bit inverse command) |
| Bit Order | LSB first |
| Repeat | Special repeat code (9 ms burst + 2.25 ms space + 562.5 us burst) |

**Timing:**

| Element | Mark (Carrier ON) | Space (Carrier OFF) |
|:---|:---|:---|
| Leader | 9,000 us | 4,500 us |
| Logical 0 | 562.5 us | 562.5 us |
| Logical 1 | 562.5 us | 1,687.5 us |
| Stop bit | 562.5 us | — |
| Repeat code | 9,000 us | 2,250 us + 562.5 us burst |

**Signal diagram:**

```
NEC Protocol — Complete Transmission (example: Address 0x00, Command 0x0C)

     Leader         Address      ~Address     Command      ~Command   Stop
  ┌──────────┐   ┌──────────── ──────────── ──────────── ────────────┐
  │          │   │                                                    │
  │  9.0 ms  │   │  8 bits      8 bits       8 bits       8 bits     │
  │  mark    │   │  (0x00)      (0xFF)       (0x0C)       (0xF3)     │
──┘          └───┘                                                    └──
             4.5ms
             space

Bit encoding detail:

  Logical "0":       Logical "1":
  ┌────┐             ┌────┐
  │562 │             │562 │
  │ us │             │ us │
──┘    └────┘     ──┘    └──────────┘
       562 us                1687.5 us

NEC Repeat Code (sent while button is held):
  ┌──────────┐
  │  9.0 ms  │
  │  mark    │
──┘          └──┐  ┌──
               │  │
    2.25 ms   │  │562 us
    space     │  │
              └──┘
  Sent every 110 ms while button remains pressed
```

**Decoding example:**

```
Raw capture: Address=0x04, Command=0x08

Binary (LSB first):
Address:  0 0 1 0 0 0 0 0  → 0x04
~Address: 1 1 0 1 1 1 1 1  → 0xFB (bitwise inverse, for error checking)
Command:  0 0 0 1 0 0 0 0  → 0x08
~Command: 1 1 1 0 1 1 1 1  → 0xF7

Verification: Address XOR ~Address = 0xFF ✓
              Command XOR ~Command = 0xFF ✓
```

> The inverse bytes provide **error detection**, not security. If `Address XOR ~Address != 0xFF`, the signal was corrupted and should be discarded. Some manufacturers use **Extended NEC** where the "inverse address" is actually a second address byte, giving 16-bit addresses (65,536 unique devices).
{: .note }

### RC5 Protocol (Philips)

Developed by Philips, RC5 uses Manchester (bi-phase) encoding and is common in European electronics.

**Specifications:**

| Parameter | Value |
|:---|:---|
| Carrier | 36 kHz |
| Encoding | Manchester (bi-phase) |
| Bits | 14 (2 start + 1 toggle + 5 address + 6 command) |
| Bit Period | 1,778 us (1/36 kHz * 64 = ~1.778 ms) |
| Bit Order | MSB first |

**Manchester encoding:**

```
Manchester Encoding:

  Logical "0":         Logical "1":
  ┌────────┐           │        ┌────────
  │ mark   │           │ space  │ mark
  │ 889 us │           │ 889 us │ 889 us
──┘        └────────   └────────┘
            space
            889 us

  Transition at the CENTER of the bit period encodes the value:
    - Low→High transition at center = Logical "1"
    - High→Low transition at center = Logical "0"
```

**RC5 frame structure:**

```
┌───┬───┬───┬────────────┬──────────────────┐
│ S1│ S2│ T │  Address   │    Command       │
│ 1 │ 1 │ 1 │  5 bits    │    6 bits        │
│bit│bit│bit│            │                  │
└───┴───┴───┴────────────┴──────────────────┘
 Start bits  Toggle bit   System   Command
 (always 1)  (alternates  (0-31)   (0-63)
              each press)
```

**Toggle bit**: Alternates between 0 and 1 with each new button press. This allows the receiver to distinguish between "button held down" (repeated code with same toggle) and "button released and pressed again" (toggle bit changes). This is a feature NEC lacks.

**RC5 address assignments:**

| Address | Device |
|:---|:---|
| 0 | TV 1 |
| 1 | TV 2 |
| 5 | VCR 1 |
| 6 | VCR 2 |
| 16 | CD |
| 17 | Audio Tuner |
| 18 | Audio Recorder |
| 20 | CD Recorder |
| 23 | AUX |

---

### RC6 Protocol (Philips, Improved)

An evolution of RC5 with more address and command bits, also Manchester encoded. Used in Microsoft MCE (Media Center Edition) remotes.

**Key differences from RC5:**

| Feature | RC5 | RC6 |
|:---|:---|:---|
| Total bits | 14 | 21+ (Mode 0: 21 bits) |
| Address bits | 5 | 8 |
| Command bits | 6 | 8 |
| Leader | None | 2,666 us mark + 889 us space |
| Toggle | Simple bit | Trailer bit with 2x bit time |
| Carrier | 36 kHz | 36 kHz |

---

### Sony SIRC Protocol

Used by Sony devices. Comes in three variants based on the number of bits.

**Specifications:**

| Parameter | 12-bit | 15-bit | 20-bit |
|:---|:---|:---|:---|
| Carrier | 40 kHz | 40 kHz | 40 kHz |
| Encoding | Pulse Width | Pulse Width | Pulse Width |
| Command bits | 7 | 7 | 7 |
| Address bits | 5 | 8 | 13 |
| Bit Order | LSB first | LSB first | LSB first |
| Repeat | 3x minimum | 3x minimum | 3x minimum |

**Timing:**

| Element | Mark (Carrier ON) | Space (Carrier OFF) |
|:---|:---|:---|
| Leader | 2,400 us | 600 us |
| Logical 0 | 600 us | 600 us |
| Logical 1 | 1,200 us | 600 us |

```
Sony SIRC Pulse Width Encoding:

  Logical "0":           Logical "1":
  ┌──────┐               ┌────────────┐
  │600 us│               │  1200 us   │
  │ mark │               │   mark     │
──┘      └──────┘     ──┘            └──────┘
         600 us                       600 us
         space                        space
```

> Sony transmits each command frame a minimum of **3 times** (45 ms frame period). The receiver only acts on the command if it receives at least 2 matching frames. This provides noise immunity. When capturing Sony IR signals, you may see 3+ identical frames in your capture — this is normal.
{: .note }

---

### Samsung Protocol

Samsung uses a modified NEC-like protocol with a different leader timing.

| Parameter | Value |
|:---|:---|
| Carrier | 38 kHz |
| Encoding | Pulse Distance |
| Bits | 32 (8-bit address + 8-bit address repeat + 8-bit command + 8-bit inverse command) |
| Leader | 4,500 us mark + 4,500 us space |
| Logical 0 | 560 us mark + 560 us space |
| Logical 1 | 560 us mark + 1,690 us space |

> Unlike NEC, Samsung sends the **address byte twice** (not the inverse). The second 8 bits are a copy of the first 8 bits. The command uses standard inverse verification like NEC.
{: .note }

---

### Panasonic Protocol

Panasonic uses a 48-bit protocol — significantly longer than most.

| Parameter | Value |
|:---|:---|
| Carrier | 37 kHz |
| Encoding | Pulse Distance |
| Bits | 48 (16-bit manufacturer + 4-bit genre + 4-bit genre2 + 8-bit data + 8-bit sub-device + 8-bit function) |
| Leader | 3,456 us mark + 1,728 us space |
| Logical 0 | 432 us mark + 432 us space |
| Logical 1 | 432 us mark + 1,296 us space |

The 48-bit payload provides a large command space but makes transmission slower.

---

### Other Common Protocols

| Protocol | Manufacturer | Bits | Carrier | Key Feature |
|:---|:---|:---|:---|:---|
| **Sharp** | Sharp | 15 (48 with expansion) | 38 kHz | Check bits, expansion code |
| **JVC** | JVC | 16 (8 addr + 8 cmd) | 38 kHz | No leader pulse on repeats |
| **Denon** | Denon/Marantz | 15 | 38 kHz | Frame + inverse frame pair |
| **LG** | LG | 28 (8 addr + 16 cmd + 4 checksum) | 38 kHz | XOR checksum |
| **Mitsubishi** | Mitsubishi (AC) | Variable (>100 bits) | 38 kHz | Full AC state in each transmission |
| **Daikin** | Daikin (AC) | Variable (>200 bits) | 38 kHz | Very long frames for AC state |
| **Whynter** | Various AC | 32 | 38 kHz | Similar to NEC timing |

### Raw IR

When a protocol cannot be identified, Bruce firmware captures the signal as **raw IR** — a sequence of alternating mark and space durations in microseconds.

```
Raw IR Example:
+9000 -4500 +560 -560 +560 -1690 +560 -560 +560 -1690
+560 -560 +560 -560 +560 -1690 +560 -560 +560 -560
+560 -1690 +560 -1690 +560 -560 +560 -560 +560 -560 ...

Format:
  +XXXX = mark (IR LED ON) for XXXX microseconds
  -XXXX = space (IR LED OFF) for XXXX microseconds
```

Raw capture is the universal fallback — it captures any IR signal regardless of protocol, but produces larger files and may be less reliable on replay due to timing precision.

> Always try protocol-specific capture first. Raw IR captures are larger, less portable, and more sensitive to timing jitter. Only use raw capture for devices with unknown or proprietary protocols.
{: .tip }

## IR Signal Capture with the T-Embed

The T-Embed CC1101 has a built-in IR receiver (typically a TSOP-series demodulator tuned to 38 kHz) and an IR transmitter LED.

### Step-by-Step Capture Process

1. **Navigate** to the IR menu: **Main Menu** > **Infrared** (or **IR**).
2. **Select** **Receive** / **Capture** / **Record** (wording varies by firmware version).
3. **Aim** your remote control directly at the T-Embed's IR receiver (usually located near the top edge of the device).
4. **Press and release** a single button on your remote. Hold it briefly (about 1 second).
5. The T-Embed displays the captured signal:
   ```
   ┌──────────────────────────────────────┐
   │ IR Signal Captured!                  │
   │                                      │
   │ Protocol: NEC                        │
   │ Address:  0x04                       │
   │ Command:  0x08                       │
   │ Bits:     32                         │
   │ Raw Length: 67 samples               │
   │                                      │
   │ [OK] Save  [B] Discard  [A] Replay  │
   └──────────────────────────────────────┘
   ```
6. **Save** the signal — give it a descriptive name (e.g., `TV_Power`, `AC_Cool_22C`).

### Understanding the Decoded Output

| Field | Meaning |
|:---|:---|
| **Protocol** | Identified encoding (NEC, RC5, Sony, Samsung, etc.) or "RAW" if unknown |
| **Address** | Device identifier — distinguishes your TV from your neighbor's TV |
| **Command** | Button function — Power, Volume Up, Channel 3, etc. |
| **Bits** | Total bits in the frame (32 for NEC, 12/15/20 for Sony, etc.) |
| **Raw Length** | Number of mark/space pairs captured (useful for raw mode) |

### Capturing Multiple Buttons

To build a complete remote profile:

1. Capture the first button (e.g., Power).
2. Save it with a clear name.
3. Return to the capture screen.
4. Capture the next button (e.g., Volume Up).
5. Repeat for all desired buttons.

**Recommended buttons to capture for a TV remote:**

| Priority | Buttons |
|:---|:---|
| Essential | Power, Volume Up, Volume Down, Mute |
| Important | Channel Up, Channel Down, Input/Source |
| Useful | Menu, OK/Select, Navigation (Up/Down/Left/Right), Back/Return |
| Nice to have | Number keys (0--9), Home, Settings, Guide |

## IR Signal Replay

### Loading and Transmitting Saved Signals

1. Navigate to **IR** > **Transmit** / **Send** / **Replay**.
2. Browse the file system to find your saved signal.
3. Select the signal file.
4. **Aim** the T-Embed's IR LED at the target device.
5. Press **OK** / confirm to transmit.
6. The target device should respond as if the original remote was used.

### Range and Angle Considerations

```
Transmission Cone:

          T-Embed IR LED
               │
              ╱│╲
             ╱ │ ╲
            ╱  │  ╲
           ╱   │   ╲
          ╱ 15-│-15 ╲
         ╱  deg│deg  ╲
        ╱      │      ╲
       ╱───────┼───────╲
      ╱ Effective Range  ╲
     ╱    (1-5 meters)    ╲
    ╱                      ╲
```

| Factor | Effect | Recommendation |
|:---|:---|:---|
| Distance | Signal weakens with distance squared | Stay within 3 m for reliable operation |
| Angle | Off-axis reduces intensity | Point directly at receiver, +-15 degrees |
| Ambient IR | Sunlight overwhelms receiver | Avoid direct sunlight on target's IR receiver |
| Obstructions | Any solid object blocks IR | Ensure clear line of sight |
| Reflections | IR reflects off white walls/ceilings | Bounced signals work but are weaker |
| Battery level | Low battery = weaker IR LED | Keep T-Embed charged |

### Troubleshooting Failed Replays

| Symptom | Likely Cause | Fix |
|:---|:---|:---|
| No response at all | Wrong aim, out of range | Move closer, aim directly at IR receiver on target device |
| Intermittent response | Marginal signal, interference | Reduce distance, block ambient IR sources |
| Wrong action triggered | Wrong signal file selected | Verify you selected the correct saved signal |
| Works on capture device but not identical model | Different IR codes by region/model | Recapture from the specific remote for that unit |
| Captured as RAW but replay fails | Timing jitter in raw capture | Recapture, or try multiple captures and use the cleanest one |
| AC unit doesn't respond | AC protocols transmit full state | Capture the complete state, not just a single button |

> **AC units are special.** Unlike TVs that receive simple commands (power toggle, volume up), air conditioners typically transmit the **entire desired state** in every IR frame — temperature, mode, fan speed, swing, timer, all in one long packet (100--200+ bits). This means you cannot simply capture "temperature up" — you must capture the complete command for each specific state you want.
{: .warning }

## TV-B-Gone Functionality

### What Is TV-B-Gone?

TV-B-Gone is a concept (originally a standalone device by Mitch Altman) that cycles through a database of **TV power-off codes** for virtually every TV manufacturer. It sends one code after another until it hits the right one and turns off the target TV.

### How It Works

```
TV-B-Gone Sequence:

  Start
   │
   ├── Send Samsung TV power code ─── wait 50ms
   ├── Send LG TV power code ──────── wait 50ms
   ├── Send Sony TV power code ─────── wait 50ms
   ├── Send Panasonic TV power code ── wait 50ms
   ├── Send Philips TV power code ──── wait 50ms
   ├── Send Toshiba TV power code ──── wait 50ms
   ├── Send Sharp TV power code ─────── wait 50ms
   ├── Send Vizio TV power code ─────── wait 50ms
   ├── Send Hisense TV power code ──── wait 50ms
   ├── ... (100+ more manufacturer codes)
   │
   └── Cycle complete (repeat or stop)
```

The database typically includes power codes for:
- Samsung, LG, Sony, Panasonic, Philips, Toshiba, Sharp, Vizio, Hisense, TCL, Insignia, Sanyo, Magnavox, Emerson, Sylvania, Funai, Element, Westinghouse, RCA, JVC, Mitsubishi, Pioneer, Hitachi, and many more.

### Running TV-B-Gone on Bruce

1. Navigate to **IR** > **TV-B-Gone** (or **IR** > **Universal** > **TV Power Off**).
2. Press **OK** to start the cycle.
3. The T-Embed begins transmitting power codes sequentially.
4. Point at the target TV.
5. The display shows which manufacturer code is currently being transmitted.
6. When the TV turns off, you can stop the cycle (or let it continue to turn off multiple TVs).
7. Press **Back** to cancel.

> **Do not use TV-B-Gone in public spaces** — restaurants, bars, airports, hospitals, or anywhere TVs serve a legitimate purpose. Turning off TVs in emergency information displays, medical monitors, or businesses is disruptive and may be illegal (interfering with business operations, or worse, safety systems). Use this feature only on your own TVs or in controlled environments for testing.
{: .danger }

### TV-B-Gone Completion Time

Because the device must cycle through many codes:

| Database Size | Approximate Cycle Time |
|:---|:---|
| 50 codes | ~5--10 seconds |
| 100 codes | ~10--20 seconds |
| 200+ codes | ~30--60 seconds |

Most common brands (Samsung, LG, Sony) are transmitted early in the cycle for faster results.

## IR in Security Context

IR control is prevalent in environments where security matters. Understanding IR attack surfaces is important for security professionals.

### Smart Home Attack Surface

| Device | IR Vulnerability | Impact |
|:---|:---|:---|
| Smart TV | IR commands bypass smart features, authentication | Channel/input change, volume, power, access settings |
| Set-top box (cable/satellite) | Change channels, access on-demand, navigate menus | Service disruption, unauthorized purchases |
| Smart AC/HVAC | Full climate control override | Comfort disruption, energy waste, potential equipment damage |
| Smart projector | Power, source, settings | Presentation disruption |
| Sound system | Volume, input, settings | Disruption, extreme volume (hearing risk) |

### IR Replay Attacks on Security Systems

Some security-adjacent systems use IR:

**IR illuminators on security cameras:**
- Many security cameras use IR LEDs for night vision.
- An external high-power IR source aimed at the camera can **blind** it (IR flood attack).
- This does not use the T-Embed's IR transmitter (insufficient power) but is a concept to understand.

**Hotel TV systems:**
- Hotel TVs often have restricted functionality (blocked inputs, limited channels).
- The restrictions are usually software-based; IR commands for "Service Menu" or "Hotel Mode" can access hidden settings.
- Common service menu access codes:
  - Samsung: Mute, 1, 1, 9, Enter (on service remote or via IR replay)
  - LG: Hold Settings button on remote for 5+ seconds
  - Philips: 1, 2, 3, 6, 5, 4 (on RC5 remote)

> Modifying hotel TV settings may violate the hotel's terms of service and could result in charges. This information is provided for security awareness, not exploitation.
{: .warning }

**AC unit control in commercial buildings:**
- Office HVAC systems often use IR remotes for individual zone control.
- Capturing and replaying these signals can override building management system settings.
- Security implication: unauthorized climate control in server rooms could cause overheating.

**IR-based access control (rare):**
- Some legacy access control systems use IR for communication between card readers and controllers.
- Extremely rare in modern installations but occasionally found in older buildings.

### Voice Assistant IR Bridges

Many smart home setups use an IR blaster (like Broadlink RM4, SwitchBot Hub) to control IR devices via voice or app. These bridges typically:
- Receive commands via Wi-Fi / cloud.
- Transmit the corresponding IR signal.

If you compromise the network, you can control any IR device through these bridges. Conversely, if you capture the IR signals the bridge sends, you can replicate its functionality with the T-Embed.

## IR Repeater and Extender Concepts

### The Problem

IR requires line of sight. Devices in cabinets, closets, or other rooms cannot receive IR signals directly.

### Using the T-Embed as a Bridge

While the T-Embed does not have a built-in IR repeater mode, you can manually:

1. Capture the IR signal from a remote.
2. Walk to the target device (even in another room or behind a cabinet door).
3. Replay the signal at close range.

For automated IR repeating, dedicated hardware (IR extenders over Cat5, RF-to-IR bridges) is more appropriate.

### IR over Wi-Fi

A more advanced approach combines the T-Embed's IR and Wi-Fi capabilities:
- Capture IR signals with the T-Embed.
- A complementary Wi-Fi-connected IR blaster (ESP32 + IR LED) in the target room receives commands over Wi-Fi and retransmits as IR.
- This effectively creates a whole-home IR control system.

## Practical Projects

### Project 1: Clone Your TV Remote

**Objective**: Create a complete backup of your TV remote on the T-Embed.

**Materials**: T-Embed, your TV remote, microSD card.

**Steps:**
1. Create a folder on the microSD card: `/ir/my_tv/`
2. For each button on your remote:
   a. Enter IR Capture mode.
   b. Press the button on your remote.
   c. Verify the capture (protocol, address, command).
   d. Save with a descriptive name.
3. Test every saved signal against your TV.
4. Note: if your TV uses different protocols for different functions (rare but possible), capture them all.

**Success criteria**: You can control your TV entirely from the T-Embed.

### Project 2: Universal Remote for Your Home

**Objective**: Replace all IR remotes in your home with the T-Embed.

**Steps:**
1. Inventory all IR-controlled devices in your home.
2. For each device, capture the buttons you use regularly.
3. Organize into numbered/named folders.
4. Test all signals.
5. Keep the T-Embed charged and accessible.

> This is especially useful for devices with lost or broken remotes. Even if you cannot find the original remote, you may be able to find the IR codes online for your device model and program them manually.
{: .tip }

### Project 3: AC Controller

**Objective**: Create a set of pre-configured AC states for one-button climate control.

**Steps:**
1. Using your AC remote, set the AC to your desired "Cool 22C" setting.
2. Press any button to transmit the full state — capture it on the T-Embed.
3. Save as `cool_22.ir`.
4. Repeat for each configuration:
   - `cool_24.ir` — energy saving cooling
   - `heat_20.ir` — light heating
   - `heat_22.ir` — comfortable heating
   - `dry_mode.ir` — dehumidifier
   - `fan_only.ir` — no heating/cooling
   - `off.ir` — turn off

**Objective**: Determine the protocol and codes used by an unknown IR remote.

**Steps:**
1. Enter IR Capture mode.
2. Press each button on the unknown remote.

5. Build a map of the remote's button layout and corresponding codes.

This is useful for:
- Reverse-engineering proprietary remote protocols.

- Creating replacement remotes for devices with discontinued remotes.

## Troubleshooting IR

### Capture issues

| Problem | Cause | Solution |
|:---|:---|:---|
| No signal detected | Remote out of range, wrong aim | Point remote directly at T-Embed receiver, < 30 cm |
| "Unknown protocol" / RAW capture | Proprietary or uncommon protocol | RAW capture still works for replay — save and test it |
| Inconsistent captures | Button bounce, timing issues | Press and hold the button firmly for 1 second; capture multiple times and compare |
| Captures work for one remote but not another | Different carrier frequencies (36 kHz vs 38 kHz vs 40 kHz vs 56 kHz) | Some receivers are less sensitive at non-38 kHz frequencies; try very close range |
| Remote has no IR? | Some "remotes" use Bluetooth or RF, not IR | Test by looking at the remote's LED through a phone camera — if you see a flash, it is IR; if not, it may be RF/BLE |

> **Quick test for IR**: Open your phone's front-facing camera (rear cameras often have IR filters). Point the remote at the camera and press a button. If you see a purple/white flash from the remote's LED, it uses IR. If you see nothing, the remote may use RF or Bluetooth.
{: .tip }

### Replay issues

| Problem | Cause | Solution |
|:---|:---|:---|
| Signal replays but device does not respond | Wrong device, wrong aim | Confirm you are aiming at the correct device's IR receiver |
| Works at 1 m but not 3 m | T-Embed IR LED is weaker than original remote | Move closer; the original remote typically has a more powerful IR LED |
| Works once then stops | Device entered a different state | Some devices require specific command sequences; recapture |
| AC unit ignores signal | Incomplete state capture | AC units require full state transmission — recapture the complete desired state |
| Signal timing is off (raw mode) | Timing jitter in capture/replay | Recapture multiple times; use protocol-decoded capture instead of raw if possible |

### General IR tips

> **IR LED power**: The T-Embed's built-in IR LED is typically less powerful than a dedicated TV remote's LED. For maximum range, aim precisely and reduce the distance. Adding an external IR LED circuit can extend range significantly if your firmware and hardware support it.
{: .note }

> **Carrier frequency mismatch**: Most IR receivers are tuned to 38 kHz but will accept a range (36--40 kHz) with reduced sensitivity. If a device uses 56 kHz (Bang & Olufsen), the standard 38 kHz receiver will have significantly reduced range. Bruce firmware typically transmits at 38 kHz or matches the detected frequency.
{: .note }

> **Multi-device interference**: If you have multiple IR-controlled devices in the same room (e.g., TV and soundbar), they may share similar protocols. Point precisely at only the intended device's IR receiver to avoid unintended actions on nearby devices.
{: .tip }
